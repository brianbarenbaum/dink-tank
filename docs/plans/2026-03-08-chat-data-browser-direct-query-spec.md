# Chat Data Browser Direct Query Spec

Date: 2026-03-08
Owner: Application Team
Status: Proposed implementation spec

## 1. Objective

Add a production-grade `Data Browser` to the chat sidebar that lets users surface common Cross Club data directly inside the chat transcript without using the LLM.

This feature exists to solve a specific product problem:

- users frequently want structured, repeatable queries such as standings, schedules, roster lists, and team summaries
- the current Cross Club site is not a preferred browsing experience
- the chat interface should remain the primary surface for both conversational answers and structured data exploration

The Data Browser is not a separate dashboard, page, or reporting section. It is a scoped query launcher embedded in the chat experience.

The user flow is:

1. Choose context in the sidebar tree.
2. Click a leaf node.
3. Insert a direct-query result card into the transcript.
4. Interact with that card in place through pagination, sorting, refresh, collapse, and removal.

## 2. Product Principles

The implementation must preserve these UX principles:

- one primary surface: all results render in the chat transcript
- source clarity: direct-query cards must be visually distinct from assistant answers
- context first: the sidebar tree scopes the query before execution
- fast common paths: leaf clicks execute immediately with no confirmation step
- predictable behavior: sidebar clicks create new cards; controls inside a card mutate that card in place
- no hidden AI: direct-query results must not be presented as if they were generated by the assistant

## 3. Goals

- expose high-frequency Cross Club queries without LLM latency or hallucination risk
- keep the chat interface as the only result surface
- make direct-query results more readable than the source site
- preserve transcript history by allowing repeated leaf clicks to create repeated cards
- support structured browsing on desktop and mobile

## 4. Non-Goals

This spec does not include:

- dedicated standings pages or schedule pages
- arbitrary user-authored SQL
- player nodes in the tree
- team search inside the tree in v1
- automatic implicit LLM grounding from every direct-query card in v1
- persistent transcript/card state across full page reloads

## 5. Information Architecture

The Data Browser is a hierarchical tree embedded inside the chat sidebar.

### 5.1 Tree Structure

```text
Season
  Division
    Players (leaf)
    Standings (leaf)
    Teams
      Team A
        Overview (leaf)
        Players (leaf)
        Schedule (leaf)
      Team B
        Overview (leaf)
        Players (leaf)
        Schedule (leaf)
```

### 5.2 Tree Rules

- multiple branches may remain expanded at once
- player names are never shown as tree nodes
- team search is not included in v1
- team lists are considered small enough to browse visually
- leaf nodes are action nodes, not navigation nodes
- clicking a leaf immediately runs a query

### 5.3 Expansion and State Memory

Within the current SPA session:

- expanded seasons remain expanded until collapsed
- expanded divisions remain expanded until collapsed
- expanded team branches remain expanded until collapsed
- sidebar state should remain intact while the user switches between cards or sends chat messages

On a full reload:

- expansion state may reset in v1
- no localStorage persistence is required for tree state in v1

## 6. Sidebar UX Spec

### 6.1 Placement

The Data Browser lives in the current chat sidebar experience, which is rendered through the chat-tab sidebar content used by the routed shell.

Primary implementation target:

- `src/features/chat/components/ChatSidebarContent.vue`

It should not be implemented only in the older standalone `ChatSidebar.vue` path, because the routed application shell currently uses `ChatSidebarContent.vue`.

### 6.2 Visual Behavior

- `Data Browser` remains a named sidebar section
- the section should feel like a structured explorer, not a placeholder accordion
- branch rows use a chevron and are clearly non-executing
- leaf rows use a different icon or affordance to communicate “run query”
- indentation must be visually clear at every level
- hover, focus, and active states must differentiate branch expansion from leaf execution

### 6.3 Default State

- desktop: `Data Browser` expanded by default on first visit
- mobile: sidebar drawer closed by default; `Data Browser` expanded when the sidebar drawer opens if it was last open in this session

### 6.4 Loading Strategy

- seasons load when chat sidebar content mounts
- divisions load lazily when a season expands
- teams load lazily when the `Teams` branch under a division expands
- leaf clicks never block on extra modal steps

### 6.5 Empty/Error States in Tree

- branch loading state shows skeleton rows or muted loading text
- empty branch state shows concise text such as `No teams found`
- branch load failures show inline retry affordances such as `Retry`

## 7. Query Interaction Model

### 7.1 Leaf Click Behavior

Every leaf click performs the following:

1. build a typed direct-query request from the selected path
2. append a loading direct-query card to the transcript immediately
3. execute a backend direct-query endpoint
4. hydrate the loading card with success, empty, or error state
5. auto-scroll transcript to the new card

### 7.2 New Card vs In-Place Mutation

This distinction is required and must remain consistent:

- sidebar leaf click: always creates a new card
- clicking the same leaf twice: creates two separate cards
- pagination inside a card: updates the existing card in place
- sorting inside a card: updates the existing card in place
- filtering inside a card: updates the existing card in place
- refresh inside a card: updates the existing card in place

This preserves a clean mental model:

- tree actions create transcript events
- card controls change view state for that transcript item

## 8. Transcript Model

The transcript must support more than plain user/assistant text.

### 8.1 Required Transcript Item Types

Recommended discriminated union:

- `user_message`
- `assistant_message`
- `direct_query_card`
- optional future `system_message`

Recommended shape:

```ts
type TranscriptItem =
  | UserMessageItem
  | AssistantMessageItem
  | DirectQueryCardItem;
```

### 8.2 Direct Query Card Shell

Every direct-query card must include:

- `Direct query` source label
- result title
- breadcrumb path
- timestamp of fetch
- state region for loading/success/empty/error
- actions row

Recommended actions:

- `Refresh`
- `Collapse`
- `Remove`
- optional future `Ask Chat About This`

### 8.3 Transcript Visual Distinction

Direct-query cards must not look like assistant speech bubbles.

They should read as structured system results:

- larger card container
- stronger borders or panel treatment
- title and breadcrumb header
- table/list/summary interior depending on card type

## 9. Supported Leaf Types

The sidebar supports five query leaf types in v1.

### 9.1 Division Players

Path:

- `Season -> Division -> Players`

Purpose:

- show a paginated list of players in the selected division

Default ordering:

- by player rank ascending

Recommended source views:

- `public.vw_player_stats_per_season`
- optional join to `public.vw_player_team` if roster metadata is needed

Recommended default page size:

- 20

Recommended columns:

- rank
- player name
- team
- record or wins/losses
- win rate
- DUPR if available

### 9.2 Division Standings

Path:

- `Season -> Division -> Standings`

Purpose:

- show a paginated list of teams in the selected division

Default ordering:

- by team rank ascending

Recommended source view:

- `public.vw_team_standings`

Recommended default page size:

- 20

Recommended columns:

- rank
- team name
- record
- win percentage
- pod/place if available
- points for/against or differential if available

### 9.3 Team Overview

Path:

- `Season -> Division -> Teams -> Team -> Overview`

Purpose:

- show a readable team snapshot rather than a table

Recommended source view:

- `public.vw_team_standings`

Presentation:

- summary block, not a paginated table

Recommended fields:

- team name
- division
- season
- rank
- pod/place
- wins
- losses
- win percentage
- points for
- points against
- point differential
- any other existing high-value team summary metrics already exposed by the standings view

### 9.4 Team Players

Path:

- `Season -> Division -> Teams -> Team -> Players`

Purpose:

- show the players on that team, sorted by rank

Recommended source views:

- `public.vw_player_stats_per_season`
- optional join to `public.vw_player_team`

Default ordering:

- rank ascending

Recommended page size:

- 12 or 20

Recommended columns:

- rank
- player name
- win rate
- record
- DUPR if available
- captain/sub flag if available

### 9.5 Team Schedule

Path:

- `Season -> Division -> Teams -> Team -> Schedule`

Purpose:

- show the team schedule in chronological order

Recommended source view:

- `public.vw_team_matches`

Presentation:

- structured row list or schedule cards, not necessarily a compact grid table

Default ordering:

- week ascending, scheduled time ascending

Recommended row fields:

- week
- date/time
- opponent
- home/away indicator
- result or status
- score summary if available

## 10. Result Card UX by Type

### 10.1 Table Cards

Applies to:

- division players
- division standings
- team players

Behavior:

- compact dense table optimized for scanning
- sticky card header
- in-card pagination controls at bottom
- optional sort toggles for selected columns
- empty state renders inside the card body, not as a separate transcript item

### 10.2 Summary Cards

Applies to:

- team overview

Behavior:

- non-tabular key metrics layout
- prominent top summary row for rank/record/win percentage
- secondary metrics in grouped stat blocks
- no pagination

### 10.3 Schedule Cards

Applies to:

- team schedule

Behavior:

- chronological list of match rows
- each row prioritizes opponent, date/time, and result
- use stronger visual hierarchy than a spreadsheet table
- if pagination is needed, keep it inside the card

## 11. Direct Query Card States

Every card must support:

- `loading`
- `success`
- `empty`
- `error`

### 11.1 Loading State

- card appears in transcript immediately after leaf click
- header is already visible
- body shows skeleton rows or summary placeholders
- actions may be disabled while loading

### 11.2 Empty State

- show concise message like `No schedule entries found for this scope`
- retain header and breadcrumb so the scope is obvious
- keep `Refresh` and `Remove`

### 11.3 Error State

- show concise inline error text
- retain breadcrumb and title
- show `Retry` and `Remove`
- do not convert backend errors into assistant prose

## 12. Recommended Backend API Shape

Use typed, authenticated worker endpoints. Do not use the SQL agent. Do not use LLM routing.

### 12.1 Context Endpoints

Recommended endpoints:

- `GET /api/data-browser/context/seasons`
- `GET /api/data-browser/context/divisions?seasonYear=...&seasonNumber=...`
- `GET /api/data-browser/context/teams?seasonYear=...&seasonNumber=...&divisionId=...`

These endpoints exist only to populate the sidebar tree.

### 12.2 Query Endpoint

Recommended endpoint:

- `POST /api/data-browser/query`

Recommended request shape:

```json
{
  "queryType": "division_players",
  "scope": {
    "seasonYear": 2025,
    "seasonNumber": 3,
    "divisionId": "uuid",
    "divisionName": "3.5",
    "teamId": null,
    "teamName": null
  },
  "viewState": {
    "page": 1,
    "pageSize": 20,
    "sortKey": "rank",
    "sortDirection": "asc"
  }
}
```

### 12.3 Query Types

Allowed values:

- `division_players`
- `division_standings`
- `team_overview`
- `team_players`
- `team_schedule`

No free-form query types are allowed.

### 12.4 Recommended Response Shape

```json
{
  "queryId": "uuid-or-stable-id",
  "queryType": "division_players",
  "breadcrumb": ["2025 S3", "3.5", "Players"],
  "title": "Division Players",
  "fetchedAt": "2026-03-08T16:00:00.000Z",
  "page": 1,
  "pageSize": 20,
  "totalRows": 48,
  "totalPages": 3,
  "sortKey": "rank",
  "sortDirection": "asc",
  "payload": {
    "columns": [...],
    "rows": [...]
  }
}
```

For summary cards, `payload` may be an object instead of rows/columns.

## 13. Backend Query Rules

- all data-browser queries must be implemented as typed repository functions
- no query may pass through the LLM
- no endpoint may accept raw SQL from the client
- all queries must remain within explicit, approved relations and application-owned SQL
- all endpoints remain protected by the current worker auth boundary

Recommended source views:

- `public.vw_player_stats_per_season`
- `public.vw_player_team`
- `public.vw_team_standings`
- `public.vw_team_matches`

## 14. Frontend Architecture Recommendation

### 14.1 State Ownership

Add a dedicated direct-query state layer for the chat transcript rather than trying to force direct-query cards into the existing plain-text chat message model.

Recommended areas:

- sidebar tree state
- transcript item collection
- direct-query card state keyed by transcript item id

### 14.2 Suggested Files

Potential file additions:

- `src/features/chat/data-browser/types.ts`
- `src/features/chat/data-browser/chatDataBrowserClient.ts`
- `src/features/chat/data-browser/useDataBrowserController.ts`
- `src/features/chat/components/DataBrowserTree.vue`
- `src/features/chat/components/DirectQueryCard.vue`
- `src/features/chat/components/DirectQueryTableCard.vue`
- `src/features/chat/components/DirectQuerySummaryCard.vue`
- `src/features/chat/components/DirectQueryScheduleCard.vue`

Potential modifications:

- `src/features/chat/components/ChatSidebarContent.vue`
- `src/features/chat/components/ChatTranscript.vue`
- `src/features/chat/types.ts`
- `src/features/chat/useChatController.ts`

### 14.3 Transcript Rendering Recommendation

`ChatTranscript.vue` should render transcript items by type:

- user text bubble
- assistant text bubble
- direct query card component

This avoids cramming direct-query payloads into markdown text bubbles.

## 15. Mobile Behavior

The tree and cards must remain usable on mobile.

### 15.1 Sidebar

- mobile data browser remains in the existing sidebar drawer
- multiple branches may stay expanded
- the drawer must remain vertically scrollable
- tapping a leaf runs the query and may close the drawer automatically after insertion to reveal the card

Recommended v1 behavior:

- after a successful leaf click on mobile, close the sidebar drawer so the user sees the newly inserted card

### 15.2 Cards

- table cards should use horizontal scroll inside the card body when needed
- summary cards should stack vertically
- schedule cards should use full-width list rows
- pagination controls must remain tap-friendly

## 16. Accessibility Requirements

- branch rows must expose correct `aria-expanded` state
- leaf rows must be buttons, not fake links
- keyboard navigation must support expanding branches and executing leaves
- direct-query cards must have clear headings
- loading, empty, and error states must be announced in an accessible way
- pagination controls must use descriptive labels

## 17. Telemetry and Observability

Track the feature as a first-class product surface.

Recommended client events:

- data browser branch expanded
- data browser leaf clicked
- direct query card loaded
- direct query card failed
- direct query pagination changed
- direct query sort changed
- direct query card refreshed
- direct query card removed

Recommended backend logs:

- query type
- scoped identifiers
- duration
- row count
- status

Add Playwright coverage for:

## 19. Acceptance Criteria

This feature is complete when all of the following are true:

- users can browse `Season -> Division -> leaf` and `Season -> Division -> Teams -> Team -> leaf`
- clicking any supported leaf appends a loading card and then renders a typed direct-query card
- leaf clicks never use the LLM
- direct-query cards are visually distinct from assistant messages
- repeated leaf clicks create repeated cards
- in-card pagination/sort/refresh mutate the existing card instead of appending a new one
- team overview renders as a summary block, not a table
- players and standings render as compact paginated tables
- schedule renders as an ordered schedule-oriented list/card layout
- desktop and mobile behaviors are both verified

## 20. Recommended Implementation Order

1. Add backend context + query endpoints with typed payloads.
2. Add frontend direct-query client and typed transcript item model.
3. Replace the `Coming soon` placeholder in the chat sidebar with the tree.
4. Add direct-query card shell and one table-card implementation.
5. Add standings and players leaves.
6. Add team overview and team schedule specialized cards.
7. Add pagination/sort/refresh/remove behavior.
8. Add E2E coverage for desktop and mobile.

## 21. Future Enhancements

Out of scope for v1, but compatible with this design:

- `Ask Chat About This` action on direct-query cards
- recents/favorites above the tree
- team search if division team counts grow
- inline comparison cards
- richer chart modules inside overview cards
- card pinning or transcript grouping
